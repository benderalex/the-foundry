# Hacker Sprint #1 — анкета по проекту The Foundry

## 1. Pitch одной фразой

The Foundry — это «фабрика фичей»: оркестратор, который превращает GitHub issue в изолированную работу агента, проверенный diff и готовый PR с наблюдаемым, перезапускаемым пайплайном.

---

## 2. Что сделано помимо ванильного `task → implement → PR`

Добавлен явный FSM-пайплайн: fetch → context → plan → implement → verify → pr → done. Это делает процесс не магическим вызовом LLM, а управляемой машиной состояний, которую можно продолжать после рестарта и отлаживать по стадиям.

Есть отдельная стадия context: оркестратор анализирует репозиторий, языки, манифесты, релевантные файлы, тестовые команды и память прошлых задач. Это снижает шанс, что агент начнёт работать вслепую.

Есть стадия plan до реализации. Если задача неоднозначна, агент может вернуть NEED_VERIFICATION, после чего пайплайн блокируется и публикует вопросы в issue. Это защита от уверенной реализации неправильной задачи.

Реализация идёт в отдельном git worktree на ветке foundry/task-{id}. Это изолирует задачи друг от друга и от основной ветки.

Сделан retry-loop implement → verify → implement: если тесты или LLM-review падают, следующая попытка получает отчёт предыдущей верификации и исправляет изменения.

Верификация двухуровневая: сначала детерминированные команды вроде ruff, pytest, npm test, затем LLM-review по diff. Так агент не может «словами» переубедить упавшие тесты.

Есть обработка PR feedback: отдельный runner читает requested changes, комментарии и failing CI checks, после чего агент дорабатывает ту же PR-ветку.

Есть observability: append-only event log в SQLite, FastAPI, SSE и React UI с live-потоком действий агента, стадиями, статусами, input/output стадий, токенами и событиями tool/text/thinking.

Есть pluggable agent backends: stub, claude_cli, codex_cli, opencode_cli. Это позволяет менять исполнителя без переписывания оркестратора.

Есть safety-слой: scrubbed env для subprocess, safe agent mode, запрет опасных команд, sanity check перед PR, лимит изменённых файлов, checkpoints diff перед попытками.

Есть repo memory: после успешных задач сохраняются touched files, verify commands и common failures, чтобы следующие задачи получали контекст накопленного опыта по репозиторию.

---

## 3. На каких задачах тестировали и под что заточен

**Тестировали на:**
- Sandbox-репо с искусственными issue (добавить функцию, починить баг, обновить README) — основной полигон.
- Реальном self-hosting: часть фич самой Foundry (mostly observability/UI и pipeline-fixes) была сделана через сам Foundry с `claude_cli` backend.
- Smoke-тесты через `scripts/add-and-process.sh` и `stub`-агента (детерминированный — добавляет строку в README) — гонятся в CI без сети.
- Кросс-языковом auto-detect verify: Python (pytest+ruff), TypeScript (npm test), Rust (cargo test), Go (go test).

**Под какие задачи заточен в первую очередь:**
- Локальный bugfix или небольшая фича в **одном** репозитории.
- Проекты с **существующими автотестами** — двухуровневый verify опирается на них.
- Issue с чётким описанием и acceptance criteria.
- Изменения, укладывающиеся в **≤40 файлов** (`MAX_FILES_PER_PR`).

**Поддерживается с оговорками:** рефакторинг (тесты ловят не всё), написание новых тестов (нет валидации качества), документация (нет детерминированного verify для prose).

**Плохо/не поддерживается:** UI-задачи с ручной проверкой, multi-repo, deploy-зависимая верификация (staging-интеграции), крупные рефакторинги.

## 4. Что бы сделали за ещё одну неделю

В порядке приоритета:

1. **Router-узел и intent-цикл (см. IDEAS.md).** Сейчас триггер — только labeled GitHub issue. Хочется одну точку входа для разных событий: PR-коммент → отдельный verify-цикл, Discord-сообщение → intent → research → создание issue, chat → диалог. Это выводит из режима «issue-first» в режим «события первичны». Скорее всего на LangGraph, чтобы не плодить самописный FSM.
2. **Сбор и отображение стоимости.** Token count и `cost_usd` уже пишутся в `task_events`, но не агрегируются по задаче и не показываются в UI. Без этого нельзя принимать решения «гонять Sonnet или Haiku на этой задаче». Это самая дешёвая фича с самым большим product-эффектом.
3. **Параллельность с явными гарантиями.** Сейчас pipeline строго последовательный (намеренно — для простоты). Минимум — несколько задач в параллель на разных worktree'ах с лимитом и кнопкой «отменить» в UI. Без отмены параллельность опасна, поэтому stop/cancel — обязательная часть.
4. **Tester/UX роль:** перед PR агент-«тестер» должен приложить скриншот UI или curl-вывод API, доказывающий, что фича работает. Без этого reviewer'у в PR приходится верить тестам на слово.
5. **Acceptance proof на этапе плана.** Планировщик кроме плана пишет «как мы поймём, что сделано» — конкретные команды/проверки. VERIFY гонит ровно это, а не auto-detected тесты.

Почему именно этот порядок: (1) разблокирует новые типы задач, (3-4) повышают качество выходного PR, (2) — единственная фича, которая ровно ничего не меняет в продукте, но без неё нельзя оптимизировать всё остальное.

## 5. Где чаще всего ломается / known failure modes

**Реально наблюдаемые:**
- **Rate-limit / 429 / 529 от LLM API.** Ловим по stderr (`rate`, `429`, `529`), retry с backoff 30/60/120s до 3 раз. Если совпало с timeout — попытка считается провальной.
- **Verify «UNCLEAR».** LLM-ревьюер вернул не `PASS` и не `FAIL: ...` — задача уходит в BLOCKED, требует человека. На дешёвых моделях случается чаще.
- **`gh push` падает на pre-existing ветке `foundry/task-{id}`** (например, после ручного эксперимента) — infra failure, re-queue. В `pr-feedback` решено через `git worktree add -B branch origin/branch`.
- **Опасный shell в plan-стадии Claude.** В safe mode permission-диалог Claude иногда блокирует команду, а агент не находит обход и возвращает невалидный план. Решение — пере-prompt'ить или временно поднять permissions.
- **`opencode_cli` не стримит инкрементальные события.** UI видит только финальный `stage_finished` — observability ломается частично, не для всех бэкендов сделан стрим (TODO в `agents/opencode_cli.py`).

**Известные хрупкие места:**
- **Git-конфликты при push** не разрешаются автоматически — только re-queue.
- **Prompt injection через тело issue** — частичная защита (изоляция worktree + denylist), но если LLM послушает «выполни X» в issue body — мы не поймаем.
- **>40 файлов в diff** → sanity-check отказывает в коммите. Для крупных рефакторингов = тупик без ручного вмешательства.
- **Sandbox escape внутренних shell-вызовов** Claude/Codex в safe mode зависит от их собственных механизмов, а не от Foundry.
- **Stub vs реальный агент.** Оффлайн-тесты гоняются на stub'е и mocked CLI fixtures — это не гарантирует, что живой `claude_cli`/`codex_cli`/`opencode_cli` backend авторизован и работает. Регрессии в реальном бэкенде ловятся smoke-прогоном.
- **Параллельность.** Не реализована намеренно: «ничего не сломается, потому что нечему».

## 6. Что сработало настолько, что рекомендуем переиспользовать

- **Линейный FSM с persisted `current_stage` в SQLite.** Перезапуск процесса не теряет прогресс. Не понадобился ни Temporal, ни LangGraph, ни Celery. Простейшая схема, которая отлично переживает crashes. Для MVP оркестратора — однозначно начинать с этого.
- **Append-only event log + SSE.** UI и pipeline — разные процессы, читающие одну БД. Никаких message queue, никаких WebSocket-комплексностей; SSE с `Last-Event-ID` решает refresh/reconnect бесплатно.
- **Двухуровневый verify (детерминированный → LLM).** Дешёво и поверочно: 80% фейлов отлавливаются `pytest`/`ruff` без единого LLM-вызова. LLM-ревьюер работает только если есть что ревьюить. Это шаблон, который должен быть в любом code-orchestrator'е.
- **Git worktree per task.** Бесплатная изоляция, нативная для git, не требует контейнеров. Failed worktree не удаляется — диагностика становится тривиальной.
- **Pluggable agent backends за тонким интерфейсом** (`src/foundry/agents/`). Смена `claude_cli` ↔ `codex_cli` ↔ `opencode_cli/deepseek` без правки workflow-кода. Полезно и для стоимости, и для тестов (offline `stub`).
- **Per-stage модели.** `haiku` для plan/verify, `sonnet`/`opus` для implement. Простой паттерн, экономит много денег.
- **Repo memory как отдельная таблица `(repo, key, value)`.** Тривиально расширяется новыми ключами, читается одним SELECT'ом, отлично работает как «long-term memory» без векторных БД.
- **`stub` agent как first-class citizen.** Offline-симуляция пайплайна плюс mocked fixtures для реальных CLI-бэкендов; текущий набор — 193 теста без сети. Без этого тесты были бы либо медленные, либо требовали бы внешней авторизации — и команда бы их выключила.
- **Checkpoint diff'ы перед каждым retry.** Дешёвая страховка (1 строка `git diff --binary > file`), которая не раз спасала полезный код из «неудачной» попытки.

## 7. Что не оправдало себя / не дало пользы

- **Слишком консервативный `MAX_FILES_PER_PR=40` без override-механизма.** Безопасно, но тупик для рефакторингов. Должен был быть конфигурируемый bypass (например, через `agent-task/large` лейбл).
- **`SAFE_AGENT_MODE=true` по умолчанию + permission-диалоги Claude.** Безопасно, но ломает часть автономности: агент периодически застревает на тривиальных shell-вызовах. На sandbox-репо `false` был бы продуктивнее по умолчанию.
- **Несколько источников правды для архитектуры.** Старые черновики быстро расходились с кодом. Канонический источник теперь один — `docs/ARCHITECTURE.md`; `DEBUG.md` оставлен как короткий runbook.
- **Sanity-check на пути (`__pycache__`, `.venv/`).** Полезно, но проще было бы делегировать `.gitignore`-логике git'а, а не поддерживать свой список запрещённых путей.
- **Polling SQLite в SSE-эндпоинте** для live-событий. Работает, но потребляет CPU на холостом ходу. Должна была быть `LISTEN/NOTIFY`-подобная схема (или хотя бы in-memory pub/sub в API-процессе) с самого начала; сейчас придётся менять.
- **Поддержка одного `TARGET_REPO` с PR в чужой репо.** Красивая идея на бумаге, но в реальной работе sandbox=target почти всегда. Сложность в коде есть, ценность — нулевая.
- Поддержка нескольких agent backends полезна, но у разных CLI разные форматы событий, resume-семантика и sandbox-флаги. Унификация оказалась дороже, чем выглядела на старте.
